# ipub 完整产品设计与实现思路

## 一、产品定位

**一句话定位：** 经本人确认后，自动整理并发布技术内容的个人出版工具。

**核心理念：** 写在你的技术笔记里，确认后自动发出去。

**不是什么：**
- 不是 AI 写作工具（不凭空生成内容）
- 不是 CMS（不管理已发布内容的生命周期）
- 不是社交媒体管理工具（不做评论回复、数据分析）

**是什么：**
- 你的原始笔记 → 可发布内容的编译器
- 多平台格式适配器
- 带审核机制的发布管道

---

## 二、目标用户

### 核心画像
- 技术博主、独立开发者、算法/工程师、研究型开发者
- 经常写实验记录、部署笔记、论文复现笔记
- 有发布意愿但因为流程摩擦而积压大量未发布内容

### 用户的原始输入
| 输入类型 | 来源 |
|---------|------|
| Markdown 笔记 | 本地文件、Obsidian |
| 云文档 | 飞书、语雀、Notion |
| 代码相关 | GitHub README/issue/commit |
| 实验记录 | 训练日志、实验报告 |
| 阅读笔记 | 论文阅读记录、书摘 |
| 碎片记录 | 踩坑日志、灵感草稿 |

## 六、各模块详细设计

### 6.1 Scanner（扫描评估器）

**职责：** 遍历笔记目录，用 LLM 评估每篇笔记的可发布价值。

**评估维度：**

| 维度 | 权重 | 说明 |
|------|------|------|
| 内容完整性 | 30% | 是否有完整的问题→方案→结果结构 |
| 技术深度 | 20% | 是否有代码、数据、原理分析 |
| 可复用性 | 20% | 其他人看了是否能从中受益 |
| 可读性 | 15% | 结构是否清晰，是否有标题分段 |
| 风险评估 | 15% | 是否有敏感信息、内部路径、不完整结论 |

**输出：**
- 0.8+ → 直接推荐进入 draft
- 0.6-0.8 → 值得发布但需要编辑
- 0.4-0.6 → 有价值但需要大幅改写
- <0.4 → 不适合发布

### 6.2 Drafter（草稿编译器）

**职责：** 把原始笔记编译为可发布的内容。

**编译原则：**
1. **保留作者声音** — 润色而不是重写
2. **保留所有技术细节** — 代码块、命令、数据原样保留
3. **改善结构** — 加摘要、理清逻辑、补充过渡
4. **去除风险** — 替换本地路径、脱敏内部信息
5. **适配平台** — 同一内容生成不同平台版本

**平台差异：**

| 平台 | 格式特点 |
|------|---------|
| Blog | Markdown + frontmatter (title, date, tags, categories) |
| CSDN | Markdown，标题要吸引点击，加摘要 |
| 知乎 | 更口语化，段落更短，适当加粗重点 |
| 飞书 | 结构清晰，适合内部分享，可加表格 |
| 公众号 | 图文并茂，段落极短，需要引导关注 |
| 小红书 | 极短，要有标签，适合碎片化分享 |

### 6.3 Reviewer（审核管理器）

**职责：** 展示草稿、支持对比、管理审核状态。

**审核界面要素：**
- 原文 vs 改写稿的 diff 对比
- 标题候选选择
- 平台选择
- 风险标记高亮
- 一键操作：approve / reject / defer

**审核原则：**
- 让用户只做 4 件事：改标题、改摘要、改少量正文、选平台
- 如果需要大幅修改 → 说明 draft 质量不够，应改进 prompt

### 6.4 Connectors（平台连接器）

**设计原则：** 每个 connector 是独立的，只负责：
1. 认证
2. 列出内容
3. 拉取内容并转为 markdown
4. （v0.2+）推送内容到平台

**已实现：**
- ✅ 飞书（文档 + 知识库，递归拉取子页面）

**待实现：**

| Connector | 认证方式 | 难度 | 优先级 |
|-----------|---------|------|--------|
| 语雀 | Token | 低 | 高 |
| GitHub | gh CLI / Token | 低 | 中 |
| Notion | Integration Token | 中 | 中 |
| CSDN | Cookie | 高 | 低 |
| 知乎 | Cookie | 高 | 低 |

### 6.5 LLM Client（LLM 调用层）

**支持多种 provider：**

```yaml
llm:
  provider: anthropic     # Claude API
  # provider: openai      # GPT-4
  # provider: local       # Ollama / vLLM
  # provider: stdin       # 手动粘贴
```

**后续扩展：**
- 支持 OpenAI（GPT-4）
- 支持本地模型（Ollama）
- 支持自定义 API endpoint

### 6.6 Style Profile（风格学习，v0.3）

**目标：** 让 LLM 逐渐学会你的写作风格。

**学习信号：**
- 你在审核时修改了哪些句子
- 你倾向选择什么样的标题
- 你常删除什么样的表达
- 你喜欢的文章结构

**风格文件：**

```yaml
# .ipub/style_profile.yaml
tone:
  - 直接
  - 技术导向
  - 少空话

structure_preference:
  - 先问题后方案
  - 代码和结果紧跟其后
  - 踩坑单独一节

avoid_phrases:
  - 众所周知
  - 值得一提的是
  - 本文将
  - 不难发现
  - 显而易见

title_style:
  - 偏好动词开头："从零复现..."、"手把手..."
  - 避免问句式标题
  - 避免过度夸张

edit_history:
  - original: "这是一个非常有趣的发现"
    edited: "这个发现很关键"
    rule: "去掉主观形容词"
```

### 6.7 内容质量增强

#### 6.7.1 "AI 味"对抗策略

技术读者对 AI 腔调极其敏感。除了 `avoid_phrases` 列表外，还需要：

**结构保留约束：**
- Prompt 中强制要求保留原始笔记的标题层级
- 不允许 LLM 自行添加"引言"、"总结"等套路段落
- 代码块、命令、输出必须原样保留

**中英文排版优化（Pangu 格式）：**
- 自动在中英文之间加空格：`使用Docker` → `使用 Docker`
- 自动在中文与数字之间加空格：`共12个` → `共 12 个`
- 这是技术内容质感的关键细节

**代码脱敏：**
- 自动检测代码中的敏感信息：`localhost`, 内部 IP, `username`, 绝对路径等
- 替换为安全占位符：`/Users/zhangsan/project` → `/path/to/project`
- 在 `risk_flags` 中标记已替换的内容

#### 6.7.2 增量扫描策略

避免每次 `scan` 都把所有笔记传给 LLM 导致费用爆炸：

**哈希校验：**
- 对每个文件内容计算 MD5，存入 `candidates.yaml`
- 只有内容变动的文件才重新评估
- 已扫描且未变动的文件直接跳过

**双阶段扫描：**
1. **轻量级初筛（本地，不调 LLM）：** 只读前 200 字 + 标题，判断是否为代码片段或完整笔记，过滤掉明显不合格的
2. **深度评估（调 LLM）：** 只有初筛通过的才进入 LLM 评分

```
100 篇笔记 → 轻量初筛 → 30 篇候选 → LLM 评估 → 10 篇推荐
                          ↑ 省掉 70% 的 LLM 调用
```

#### 6.7.3 审核体验增强

**编辑器集成：**
- `ipub review <id>` 支持通过 `$EDITOR` 环境变量调起编辑器（VS Code / Vim）
- 直接编辑 `draft.md`，保存即更新
- 支持 `--editor` 参数指定编辑器

**浏览器预览：**
- `ipub review <id> --preview` 在本地浏览器打开 Markdown 渲染页面
- 使用临时 HTML 文件，无需启动服务器

**Diff 对比：**
- `ipub review <id> --diff` 显示原文与改写稿的逐行对比
- 使用 `difflib` 生成彩色终端输出

### 6.8 Media Manager（图片管理，v0.2+）

技术笔记里往往有本地图片路径，直接发布会导致图片 404。

**职责：**
- 在 `publish` 阶段，自动检测 Markdown 中的本地图片路径
- 上传到图床（GitHub、S3、阿里云 OSS）
- 替换 Markdown 中的路径为公开 URL
- 记录图片映射关系，避免重复上传

**支持的图床：**

| 图床 | 优先级 | 说明 |
|------|--------|------|
| GitHub repo | 高 | 免费，利用 GitHub raw 链接 |
| 阿里云 OSS | 中 | 国内访问快 |
| S3 | 中 | 通用 |
| SM.MS | 低 | 免费图床，有容量限制 |

**配置：**

```yaml
media:
  image_host: github          # 图床类型
  github_repo: "wbzuo/blog-images"
  upload_on_approve: true     # 审核通过时自动上传
```

### 6.9 平台特定适配逻辑

不同平台对 Markdown 的支持和排版习惯不同，需要在导出时做针对性处理：

| 平台 | 自动处理 |
|------|---------|
| **知乎** | `###` 及以下标题降级为 `**加粗**`，知乎多级标题支持差 |
| **CSDN** | 文首自动添加 `@[TOC]` 目录，适配 CSDN 目录功能 |
| **飞书** | Markdown 转飞书文档块格式（使用飞书 API 的 block 结构） |
| **微信公众号** | 提示"缺少封面图"，生成封面图描述供 AI 绘图参考 |
| **小红书** | 自动截取前 500 字 + 标签提取，适配短内容格式 |

### 6.10 发布幂等性（v0.2）

自动发布时最怕网络超时导致重复发布：

- 在 `published.yaml` 中不仅记录 URL，还记录平台返回的 **Remote ID**
- 再次发布时先检测 ID 是否存在：
  - 存在 → 执行 `update` 而不是 `create`
  - 不存在 → 正常创建
- 每次发布操作都记录 `publish_attempt_id`，支持重试

```yaml
# published.yaml 示例
- draft_id: "001"
  platforms:
```

## 八、关键技术决策

### 8.1 为什么用文件存储而不是数据库？

- 用户是技术人员，能直接查看和编辑 yaml/md 文件
- git 友好，可以版本管理
- 零依赖，不需要额外安装数据库
- 数据量不大（几百篇笔记），文件完全够用

### 8.2 为什么 CLI 而不是 Web UI？

- 技术用户更习惯 CLI
- 开发快，一个人能维护
- 可以被其他工具（Claude Code, cron）调用
- Web UI 作为 v0.7 的增强，不是核心

### 8.3 为什么不做全自动发布？

- 用户需要保留作者身份和最终确认权
- 自动发布的内容质量不可控，会伤害个人品牌
- Human-in-the-loop 是产品的核心价值观

### 8.4 Prompt 设计原则

- 明确告诉 LLM "润色而不是重写"
- 给出 avoid_phrases 列表防止 AI 味
- 要求保留所有代码块和技术细节
- 要求标记不确定的推断
- 风格要求写在 system prompt 里

## 九、竞品分析

| 产品 | 定位 | 与 ipub 的区别 |
|------|------|---------------|
| Notion AI | 通用 AI 写作 | 不专注技术内容，不做多平台适配 |
| Hashnode | 技术博客平台 | 只是一个平台，不处理你的已有笔记 |
| Buffer | 社交媒体管理 | 面向营销，不理解技术内容 |
| Ghost | 博客平台 | 不做内容发现和编译 |
| Obsidian Publish | 笔记发布 | 只发到 Obsidian 自己的平台 |

**ipub 的差异化：**
1. 从已有笔记发现可发布内容（不是从零写）
2. 保留作者风格（不是 AI 代写）
3. 多平台格式适配（不是只发一个平台）
4. 人工审批机制（不是全自动）
5. 本地优先（不是 SaaS）
